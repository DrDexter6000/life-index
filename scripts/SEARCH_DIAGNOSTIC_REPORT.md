## 四、发现的问题

### I-1 [设计] 中文多词查询强制 OR，精确度不足
**现象**: K02 "乐乐 钢琴" 返回 7 条（与 "乐乐" 单独搜索相同），期望只有钢琴课那 1 条。
**根因**: `keyword_pipeline.py:124-129` 对 jieba 分词后的查询强制使用 `OR` 模式：
```python
if was_segmented:
    return " OR ".join(keywords), None
```
**设计意图**: 中文自然语言查询在概念上连贯，严格 AND 匹配会导致零召回。但对用户明确输入的两个独立关键词，OR 降级过于激进。
**建议**: 区分"用户主动多词输入"（倾向 AND）vs"分词扩展后的多 token"（倾向 OR）。

### I-2 [召回] min-hits 过滤规则对自然语言查询过于严格
**现象**: 5 组 WARN 中有 4 组是因为 `FTS min-hits filter` 把 FTS 候选结果全部过滤掉了：
- S01: `3 → 0 results`（要求 ≥2 of ['多久','关心','健康','运动']）
- S06: `2 → 0 results`（要求 ≥3 of ['圆圆','成长','变化','学会','新技能']）
- S08: FTS 候选被过滤（天气相关）
- S10: `1 → 0 results`（要求 ≥4 of ['时间','管理','做','好不好',...]）

**根因**: D8 规则 `max(2, ceil(non_stopword_tokens × 0.4))` 在自然语言查询 token 较多时，要求命中太多不同 token。jieba 切出的有效 token 太多（"好不好"、"有没有" 未被 stopword 过滤），拉高了阈值。
**影响**: 语义后端不可用时，自然语言查询的 FTS 降级路径召回率仅 50%。

### I-3 [召回] Stopword 过滤不充分
**现象**: S10 查询 "时间管理做得好不好？有没有决定改变什么习惯？" 切出的 token：
`['时间', '管理', '做', '好不好', '有没有', '决定', '改变', '习惯']`
其中 "做"、"好不好"、"有没有" 是纯噪音词，未被中文 stopword 列表过滤。
**根因**: 中文 stopword 列表缺少这些口语化词汇。jieba 切出细粒度 token 后，很多是语法助词。

### I-4 [提取] 关键词提取混合中英文时断裂
**现象**: S04 查询 "Life Index项目最近有哪些重要进展和里程碑？" 提取的关键词：
`['Life', 'Index项目有哪些重要进展和里程碑？']`
**根因**: jieba 无法正确识别 "Life Index" 为一个整体品牌名 + 中文后缀的混合结构，把 "Index" 后面的中文全部粘合成一个 token。

### I-5 [警告] semantic=False 时仍输出降级警告
**现象**: 所有 10 组关键词搜索（`semantic=False`）都输出了两条警告：
- `semantic_disabled: 用户通过 --no-semantic 禁用语义搜索`
- `semantic_unavailable: 语义搜索已通过 --no-semantic 禁用。`
**根因**: `core.py` 在 `semantic=False` 时主动添加 disabled 警告，同时语义管线又添加 unavailable 警告，形成重复。
**影响**: 警告噪音，对 API 调用者造成困惑。

### I-6 [已知限制] 语义后端不可用时的降级体验
**现象**: 所有 10 组语义查询都因无向量索引而降级为纯关键词搜索。
**说明**: 这是预期行为——测试环境没有 sentence-transformers 和向量索引。在生产环境中，语义搜索会提供更好的自然语言理解能力。

### I-7 [性能] 首次查询冷启动 ~900ms
**现象**: K01 首次搜索耗时 959ms（jieba 模型加载），后续查询稳定在 20-45ms。
**说明**: 一次性成本，不影响实际使用体验。

### I-8 [数据] FTS 无法搜索 frontmatter 中的 mood 值
**现象**: S05 搜索 "心情/开心/低落" 返回 0 条，但 20 篇日志的 frontmatter 中都有 mood 字段（如"充满希望"、"欣慰"、"疲惫"等）。
**根因**: FTS 索引只索引 content 字段和 title_segmented，不索引 frontmatter 中的 mood/tags 等结构化字段。情绪搜索需要走 L2 元数据过滤（通过 mood 参数），而非 FTS 内容搜索。
**说明**: 这是搜索接口设计问题，不是 bug——用户需要通过 `mood=["开心"]` 参数搜索，而非在 query 中输入。

## 六、结论

**搜索管线在关键词场景下运行稳定，10/10 全 PASS。**

自然语言搜索的 50% WARN 均由两个根因导致：
1. **语义后端不可用**（测试环境限制，非代码问题）
2. **FTS 降级路径的 min-hits 过滤过于严格**（I-2，核心问题）

在 8 周维护冻结期内，建议关注的优先级：
- **P1 (冻结期可修)**: I-5 警告噪音去重 — 改动小，风险低
- **P2 (冻结期可修)**: I-3 中文 stopword 补充 — 纯数据改动，无代码风险
- **P3 (冻结期后)**: I-1 AND/OR 智能切换、I-2 min-hits 自适应阈值、I-4 混合语言分词
