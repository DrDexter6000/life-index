## Phase 1：依赖替换 — sentence-transformers → fastembed

### 目标
将推理框架从 `sentence-transformers`（PyTorch 后端，~3-4GB）替换为 `fastembed`（ONNX Runtime 后端，~200-300MB），大幅缩小安装体积。

### 为什么换
- `sentence-transformers` 强制依赖 PyTorch（~2GB）+ transformers（~500MB）+ tokenizers 等
- 对于一个"小而美"的个人日志工具，3-4GB 的语义搜索依赖不可接受
- `fastembed`（by Qdrant）使用 ONNX Runtime，同样支持 sentence-transformers 模型，但安装体积仅 ~200-300MB
- 推理性能相当，CPU-only 场景下 ONNX 甚至更快

#### 1.2 重写 `tools/lib/semantic_search.py`

**定位文件**: `tools/lib/semantic_search.py`（641 行）

这个文件是 sentence-transformers 的封装层。需要将底层从 `SentenceTransformer` 替换为 `fastembed.TextEmbedding`。

**核心 API 对照**:

| 操作 | sentence-transformers | fastembed |
|:---|:---|:---|
| 初始化 | `SentenceTransformer(model_name)` | `TextEmbedding(model_name)` |
| 编码 | `model.encode(texts)` → `np.ndarray` | `list(model.embed(texts))` → `list[np.ndarray]` |
| 返回类型 | numpy array (直接) | generator (需 `list()` 包装) |
| 模型缓存 | `~/.cache/torch/sentence_transformers/` | `~/.cache/fastembed/` |

**关键修改点**:

1. 所有 `from sentence_transformers import SentenceTransformer` → `from fastembed import TextEmbedding`

2. 模型加载:
   ```python
   # 修改前
   self._model = SentenceTransformer(model_name, cache_folder=str(cache_dir))

   # 修改后
   self._model = TextEmbedding(model_name, cache_dir=str(cache_dir))
   ```

3. 编码调用:
   ```python
   # 修改前
   embeddings = self._model.encode(texts, show_progress_bar=False, normalize_embeddings=True)

   # 修改后
   embeddings = list(self._model.embed(texts))
   # fastembed 默认输出 L2-normalized embeddings，无需额外 normalize
   ```

4. **保留所有现有的异常处理、缓存机制、日志记录逻辑** — 只换底层调用

5. **移除所有 `try: import sentence_transformers except ImportError` 防御性导入** — fastembed 现在是核心依赖，不需要优雅降级

## Phase 2：模型替换 — all-MiniLM-L6-v2 → paraphrase-multilingual-MiniLM-L12-v2

### 目标
将嵌入模型从英文 only 的 `all-MiniLM-L6-v2` 替换为多语言的 `paraphrase-multilingual-MiniLM-L12-v2`，解决中文日志语义搜索质量极差的核心问题。

### 为什么换
- `all-MiniLM-L6-v2` 使用英文 WordPiece tokenizer，中文字符被拆分为单字 `[UNK]` 或随机子词
- 中文查询 "想念女儿" 和日志内容 "思念团团" 之间几乎无法建立语义关联
- `paraphrase-multilingual-MiniLM-L12-v2` 训练数据覆盖 50+ 语言（含中文），输出维度同为 384，无需修改向量索引结构
- 模型体积从 ~80MB 增至 ~420MB，在可接受范围内

### 操作步骤

## Phase 3：搜索架构 — 串行 → 并行双管道

### 目标
将搜索从 L1→L2→L3→(opt L4) 串行流水线改为 Pipeline A (FTS5) ∥ Pipeline B (Semantic) 并行执行 + RRF 融合。

### 架构变更

```
            ┌─────────────────────────────────────┐
            │            用户查询                   │
            └─────────┬───────────────┬─────────────┘
                      │               │
                ┌─────▼──────┐  ┌─────▼──────┐
                │ Pipeline A │  │ Pipeline B │
                │   关键词    │  │   语义     │
                │            │  │            │
                │ L1 filter  │  │  向量搜索   │
                │ L2 filter  │  │ (fastembed) │
                │ L3 FTS5    │  │            │
                └─────┬──────┘  └─────┬──────┘
                      │               │
                ┌─────▼───────────────▼──────┐
                │    RRF 融合 (k=60)          │
                │  keyword_weight / semantic_ │
                │  weight 可配置              │
                └────────────┬───────────────┘
                             │
                        最终排序结果
```

**关键设计决策**：
- L1/L2 不再是独立层级，而是 Pipeline A 的前置过滤器
- Pipeline B (语义) 独立运行，不依赖 L1/L2 的输出
- 两个 Pipeline 通过 `ThreadPoolExecutor` 并行执行
- RRF 融合已存在于 `ranking.py`，复用即可

### 操作步骤

#### 3.1 修改 `tools/search_journals/core.py`

**定位文件**: `tools/search_journals/core.py`（240 行）

这是搜索的核心协调模块。当前的 `hierarchical_search()` 函数串行执行 L1→L2→L3→L4。

**重写策略**: 将函数体重组为两个并行管道。

```python
# 修改后的 core.py 结构（伪代码）

import time
from concurrent.futures import ThreadPoolExecutor, as_completed
from typing import Any, Dict, List, Optional

from .l1_index import scan_all_indices, search_l1_index
from .l2_metadata import search_l2_metadata
from .l3_content import search_l3_content
from .semantic import search_semantic
from .ranking import merge_and_rank_results_hybrid


def hierarchical_search(
    query: Optional[str] = None,
    topic: Optional[str] = None,
    project: Optional[str] = None,
    tags: Optional[List[str]] = None,
    mood: Optional[List[str]] = None,
    people: Optional[List[str]] = None,
    date_from: Optional[str] = None,
    date_to: Optional[str] = None,
    location: Optional[str] = None,
    weather: Optional[str] = None,
    level: int = 3,
    use_index: bool = True,       # 默认值改为 True（FTS 优先）
    semantic: bool = True,         # 默认值改为 True（语义默认启用）
    semantic_weight: float = 0.4,
    fts_weight: float = 0.6,
) -> Dict[str, Any]:
    """
    双管道并行搜索

    Pipeline A: L1 索引过滤 → L2 元数据过滤 → L3 FTS5 内容匹配
    Pipeline B: 语义向量搜索
    融合: RRF (Reciprocal Rank Fusion, k=60)
    """
    result: Dict[str, Any] = {
        "success": True,
        "query_params": { ... },  # 保持不变
        "l1_results": [],
        "l2_results": [],
        "l3_results": [],
        "semantic_results": [],
        "merged_results": [],
        "total_found": 0,
        "performance": {},
    }

    start_time = time.time()

    # ── Pipeline A: 关键词搜索（L1 → L2 → L3）──
    def pipeline_keyword():
        """关键词搜索管道"""
        perf = {}

        # L1: 索引过滤
        l1_start = time.time()
        l1_results = []
        if topic:
            l1_results.extend(search_l1_index("topic", topic))
        if project:
            l1_results.extend(search_l1_index("project", project))
        if tags:
            for tag in tags:
                l1_results.extend(search_l1_index("tag", tag))
        # 去重
        seen = set()
        l1_results = [r for r in l1_results if r["path"] not in seen and not seen.add(r["path"])]
        perf["l1_time_ms"] = round((time.time() - l1_start) * 1000, 2)

        # L2: 元数据过滤
        l2_start = time.time()
        l2_response = search_l2_metadata(
            date_from=date_from, date_to=date_to, location=location,
            weather=weather, topic=topic, project=project,
            tags=tags, mood=mood, people=people, query=query,
        )
        l2_results = l2_response["results"]
        perf["l2_time_ms"] = round((time.time() - l2_start) * 1000, 2)

        # L3: FTS5 内容搜索
        l3_start = time.time()
        l3_results = []
        if query:
            # FTS5 查询（逻辑保持不变，从现有代码搬运）
            # ... 完整保留现有的 FTS5 查询逻辑 ...
            pass
        perf["l3_time_ms"] = round((time.time() - l3_start) * 1000, 2)

        return l1_results, l2_results, l3_results, perf

    # ── Pipeline B: 语义搜索 ──
    def pipeline_semantic():
        """语义搜索管道"""
        if not (semantic and query):
            return [], {}
        sem_start = time.time()
        sem_results = search_semantic(query, date_from or "", date_to or "")
        perf = {"semantic_time_ms": round((time.time() - sem_start) * 1000, 2)}
        return sem_results, perf

    # ── 并行执行 ──
    with ThreadPoolExecutor(max_workers=2) as executor:
        future_keyword = executor.submit(pipeline_keyword)
        future_semantic = executor.submit(pipeline_semantic)

        # 收集结果
        l1_results, l2_results, l3_results, kw_perf = future_keyword.result()
        semantic_results, sem_perf = future_semantic.result()

    # 填充结果
    result["l1_results"] = l1_results
    result["l2_results"] = l2_results
    result["l3_results"] = l3_results
    result["semantic_results"] = semantic_results
    result["performance"].update(kw_perf)
    result["performance"].update(sem_perf)

    # ── RRF 融合 ──
    if semantic_results:
        result["merged_results"] = merge_and_rank_results_hybrid(
            l1_results, l2_results, l3_results, semantic_results,
            query, fts_weight=fts_weight, semantic_weight=semantic_weight,
        )
    else:
        # 语义搜索无结果时退化为纯关键词排序
        from .ranking import merge_and_rank_results
        result["merged_results"] = merge_and_rank_results(
            l1_results, l2_results, l3_results, query,
        )

    result["total_found"] = len(result["merged_results"])
    result["performance"]["total_time_ms"] = round((time.time() - start_time) * 1000, 2)

    return result
```

**关键注意事项**:

1. **保留 `level` 参数的向后兼容**：当 `level=1` 或 `level=2` 时，按原逻辑提前返回（只跑 Pipeline A 的部分层级），不启动并行。只有 `level=3`（默认）时才启动双管道并行。

2. **保留 `use_index` 参数**：控制 L3 是用 FTS5 还是文件系统扫描。默认值改为 `True`。

3. **`semantic` 参数默认值改为 `True`**：语义搜索默认启用。

4. **ThreadPoolExecutor 线程安全**：SQLite FTS5 和向量索引的读操作是线程安全的（SQLite WAL mode + 只读查询）。无需加锁。

## Phase 4：Write-Through — 写入时同步更新向量索引

### 目标
在 `write_journal` 写入日志后，同步更新向量索引，消除"写入后搜不到"的延迟窗口。

### 当前问题
- `write_journal/core.py` 写入日志后更新 by-topic 索引和 FTS5 索引，但**不更新向量索引**
- 向量索引仅通过 cron 定时任务更新（daily 23:50 增量 + monthly 全量重建）
- 这意味着新写入的日志在最多 24 小时内无法被语义搜索找到

### 操作步骤

#### 4.1 在 `write_journal/index_updater.py` 中添加向量索引更新函数

**定位文件**: `tools/write_journal/index_updater.py`（209 行）

在文件末尾新增函数：

```python
def update_vector_index(journal_path: Path, data: Dict[str, Any]) -> bool:
    """
    写入后同步更新向量索引（Write-Through）

    Args:
        journal_path: 日志文件路径
        data: 日志数据（包含 title, content, abstract 等）

    Returns:
        True 如果更新成功，False 如果失败（不影响主流程）
    """
    try:
        from ..lib.vector_index_simple import get_model, get_index
        from ..lib.config import USER_DATA_DIR

        model = get_model()
        if not model.load():
            return False

        # 构建要嵌入的文本（与 build_index 保持一致）
        title = data.get("title", "")
        content = data.get("content", "")
        abstract = data.get("abstract", "")

        # 拼接用于嵌入的文本：标题 + 摘要 + 内容前 500 字
        embed_text = f"{title} {abstract} {content[:500]}".strip()
        if not embed_text:
            return False

        # 生成嵌入向量
        embeddings = model.encode([embed_text])
        if not embeddings:
            return False

        # 计算相对路径
        try:
            rel_path = str(journal_path.relative_to(USER_DATA_DIR)).replace("\\", "/")
        except ValueError:
            rel_path = str(journal_path).replace("\\", "/")

        # 更新索引
        index = get_index()
        date_str = str(data.get("date", ""))[:10]
        index.add(rel_path, embeddings[0], {"date": date_str, "title": title})
        index.save()

        return True

    except (ImportError, OSError, IOError, RuntimeError) as e:
        # 向量索引更新失败不应阻塞日志写入
        # 日志会在下次 cron 全量重建时补上
        return False
```

**注意**：确认 `build_index` 模块中用于嵌入的文本拼接逻辑，确保 `update_vector_index()` 使用相同的拼接策略。如果不同，语义搜索结果会不一致。查看 `tools/build_index/` 中的嵌入文本构建逻辑，保持一致。

#### 4.2 在 `write_journal/core.py` 中调用向量索引更新
