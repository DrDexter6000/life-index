# RFC: Index Tree Evidence Navigation

Status: Draft
Created: 2026-05-14
Updated: 2026-05-30
Classification: design-only future work; no API commitment; no version bump;
no public CLI implementation in this RFC update
Related: ADR-026, docs/ARCHITECTURE.md section 1.4, docs/API.md aggregate
section, tools/generate_index/navigation.py, tools/aggregate/claim_envelope.py,
tools/evidence, tools/dev/everos_derived_memory_spike.py

## 1. Naming Decision

The roadmap name is **Index Tree Evidence Navigation**.

"Evidence Map" may appear only as an explanatory phrase for humans, meaning
"a map of where evidence lives across root/year/month index nodes." It is not
the feature name, not a future command name, and not a schema name.

## 2. Problem

Life Index already generates a three-level Index Tree: root, year, and month.
Today it mostly serves human browsing. For long-horizon personal-log work, the
same deterministic tree can also serve as an evidence navigation substrate:
scope estimation, coarse narrowing, evidence paging, and resumable batch work.

EverOS / EverCore research adds one useful lesson: raw memory is easier to use
when rebuildable views organize it into traversal shapes such as topic,
episode, and fact. Life Index should translate that idea into its own terms,
without adopting EverOS service infrastructure or profile/persona memory as
truth.

Life Index terms:

| EverOS / EverCore pattern | Life Index translation |
|---|---|
| topic traversal | frontmatter-derived navigation signal over month/year nodes |
| episode memory | journal entry reference anchored to an Index Tree node |
| atomic fact | frontmatter-derived candidate with evidence path, not truth |
| memory graph / evidence map | Index Tree Evidence Navigation over root/year/month nodes |
| profile/persona memory | out of core; future L3 consumer may interpret evidence, but core does not |

This RFC records the roadmap without changing public CLI behavior.

## 3. Current Ground

The current tree is generated by `generate-index` from local Markdown/YAML data:

| Level | Artifact | Intended use today |
|---|---|---|
| Root | `INDEX.md` | Global map and year overview |
| Year | `Journals/YYYY/index_YYYY.md` | Month overview and metadata aggregates |
| Month | `Journals/YYYY/MM/index_YYYY-MM.md` | Entry table and month-level metadata |

Current machine-facing anchors already exist, but are deliberately narrow:

- `tools/generate_index/navigation.py` has internal read-only helpers for
  `IndexNode`, `enumerate_index_nodes`, freshness checks, month refs, and range
  refs.
- `tests/unit/test_index_tree_navigation.py` covers node ids, freshness, topic
  aggregation, and month ref shape.
- Aggregate evidence items can carry `index_node_ref` pointing back to month
  nodes.
- `docs/API.md` states `generate-index --json` is an array-shaped public
  contract and does not expose a full Index Tree API.

Therefore this roadmap must preserve the current public shape:

- no change to `generate-index --json`;
- no new public command in Phase 1;
- no change to `search` default ranking;
- no version bump for a private research prototype.

## 4. Before / After Architecture

### Before

```text
Journals/*.md
  ├─ YAML frontmatter
  └─ Markdown body
        │
        ▼
generate-index
        │
        ├─ INDEX.md
        ├─ Journals/YYYY/index_YYYY.md
        └─ Journals/YYYY/MM/index_YYYY-MM.md

search / aggregate / analyze
        │
        └─ read journals and existing indexes opportunistically
```

The Index Tree is mainly a human-readable browsing artifact. Machine-readable
navigation exists only as internal helpers and small `index_node_ref` hooks.

### After

```text
Journals/*.md
  ├─ deterministic frontmatter fields
  └─ Markdown body
        │
        ▼
shared deterministic index builder
        │
        ├─ Markdown Index Tree
        │    ├─ INDEX.md
        │    ├─ Journals/YYYY/index_YYYY.md
        │    └─ Journals/YYYY/MM/index_YYYY-MM.md
        │
        └─ JSON sidecar / manifest
             ├─ root/year/month node ids
             ├─ entry refs and relative paths
             ├─ frontmatter-derived navigation signals
             ├─ freshness / source hashes
             └─ evidence paging cursors

L3 / dev consumers
  ├─ enumerate nodes
  ├─ page evidence by node
  ├─ run Search Shadow Mode
  └─ build Evidence Pack / Claim Envelope inputs

default search ranking
  └─ unchanged unless a later shadow-mode gate proves recall preservation
```

The long-term main path must be a shared deterministic builder that generates
Markdown and JSON sidecar data from the same source pass. The manifest must not
depend on reverse-parsing Markdown indexes as the permanent architecture.

## 5. Proposed Future Role

Index Tree Evidence Navigation should become a read-only navigation layer for
future L3 modules, not a replacement for search, aggregate, analyze, or recall.

It can provide four functions:

| Function | Value |
|---|---|
| Sitemap | Estimate work size before reading individual journals |
| Coarse narrowing | Exclude empty or obviously irrelevant years/months before deeper retrieval |
| Evidence paging | Group evidence by stable month nodes so long answers remain auditable |
| Checkpoint/resume | Let long-running jobs resume from the next unfinished year/month node |

Intended flow:

```text
user request
  -> root node: which years contain data?
  -> year nodes: which months are plausible?
  -> month nodes: which entries are candidates?
  -> journal files: read full evidence only where needed
  -> evidence pack / claim envelope: cite exact paths and node refs
```

The phrase "coarse narrowing" is important. Index nodes may reduce work, but
must not become answer truth. Full journal evidence remains the source for any
claim.

## 6. EverOS / EverCore Absorption Boundary

Useful patterns to absorb:

- Rebuildable evidence organization views are valuable.
- Topic -> episode -> fact traversal maps well to Life Index if all items keep
  evidence paths.
- Candidate views can help reflection, memoir scaffolds, and persona evidence
  bundles without becoming truth.
- Evaluation should test Add/Search/Answer/Evaluate-like stages separately:
  source collection, navigation, retrieval, claim assembly, answer synthesis.

Patterns not to absorb into core:

- service stack: MongoDB, Elasticsearch, Milvus, Redis, daemon, HTTP service;
- profile/persona memory as a truth store;
- automatic foresight memory without explicit confirmation;
- narrative, persona, relationship, therapeutic, or emotional judgment inside
  L1/L2 core;
- LLM-produced summaries as machine-trusted Index Tree facts.

Life Index may use LLM-assisted typed frontmatter as a navigation signal when
that field was produced through an explicit write-time or edit-time opt-in flow
and passed schema validation. Even then, the signal remains metadata for
navigation, not core truth about a person's identity, relationship, intent, or
psychology.

Core Index Tree truth may include only deterministic facts:

- relative journal paths;
- dates and node ids;
- counts and ranges;
- validated frontmatter fields;
- source hashes / freshness state;
- evidence references.

## 7. Conceptual Future Interfaces

These are conceptual shapes only. A future implementation must go through a
separate design and TDD plan.

### 7.1 Enumerate Index Nodes

Purpose: list deterministic year/month nodes and their metadata without reading
all journal bodies.

Conceptual shape:

```text
enumerate_index_nodes(scope, date_range, filters) -> node summaries
```

Example node summary:

```json
{
  "node_id": "month:2026-03",
  "level": "month",
  "path": "Journals/2026/03/index_2026-03.md",
  "entry_count": 12,
  "topics": {"work": 8, "life": 4},
  "freshness": "fresh"
}
```

### 7.2 Resolve Evidence Paths

Purpose: given an evidence item date or path, return its index node reference.
This generalizes the current aggregate evidence `index_node_ref` pattern.

### 7.3 Page Over Nodes

Purpose: process long histories in bounded month/year chunks.

Conceptual shape:

```text
index_node_page(start_node, direction, chunk_size) -> nodes + next_cursor
```

This would be a stable node cursor, not a database cursor and not a promise to
keep full journal content in memory.

### 7.4 Private Manifest Prototype

Phase 1 may create a private developer manifest under `tools/dev` only:

```json
{
  "schema_version": "index_tree_evidence_manifest.dev.v0",
  "command": "dev.index_tree_evidence_manifest",
  "source": {
    "builder": "prototype",
    "public_contract": false
  },
  "nodes": [
    {
      "node_id": "month:2026-03",
      "level": "month",
      "relative_path": "Journals/2026/03/index_2026-03.md",
      "entry_refs": [
        "Journals/2026/03/life-index_2026-03-14_001.md"
      ],
      "frontmatter_signals": {
        "topic_counts": {"work": 2},
        "people_counts": {"Alice": 1},
        "project_counts": {"Life Index": 1}
      },
      "freshness": "fresh"
    }
  ],
  "limitations": [
    "Private dev artifact; not a public CLI/API contract.",
    "Navigation signals are not narrative truth."
  ]
}
```

Phase 1 must not modify `generate-index --json`, search ranking, or public
docs/API.

Privacy boundary: `frontmatter_signals` keys may contain real person names,
project names, locations, or other journal-derived private labels. Any persisted
report or manifest that includes these dictionaries has the same privacy level
as the underlying journals. If this artifact is ever promoted into a public
contract, it must go through a separate privacy review before publication.

## 8. Roadmap

The roadmap proper has four phases: private manifest, shared deterministic
builder, Search Shadow Mode, and promotion decision. The current RFC update is
only a scope lock and does not count as an implementation phase.

### Phase 2 - Shared Deterministic Builder

Purpose: remove the long-term anti-pattern of reverse-parsing Markdown indexes
as the main path.

Phase 2 technical debt: the Phase 1 prototype reads both Index Tree nodes and
journal frontmatter, which creates a double-parse / drift risk between Markdown
indexes and manifest signals. The long-term path must converge on a shared
deterministic builder rather than depending on Markdown index reverse parsing.

Allowed file impact:

- Refactor within `tools/generate_index/**` to isolate a shared deterministic
  builder that returns an in-memory node model.
- Add tests under `tests/unit/test_generate_index_builder.py` or extend
  `tests/unit/test_index_tree_navigation.py`.
- Add a private JSON sidecar writer only if a separate TDD plan proves path,
  schema, and compatibility boundaries.

Forbidden file impact:

- Do not change public `generate-index --json` array shape.
- Do not remove or rename Markdown index files.
- Do not make the sidecar a public API without a new RFC/PRD.
- Do not use an LLM to populate builder truth fields.

Tests / checks:

```bash
python -m pytest tests/unit/test_index_tree_navigation.py tests/unit/test_generate_index*.py -q --basetemp .pytest_tmp/index-tree-builder
python -m black --check tools/generate_index tests/unit/test_index_tree_navigation.py
python -m flake8 tools/generate_index --count --max-complexity=40 --max-line-length=100 --show-source --statistics
git diff --check
```

Exit criteria:

- Markdown indexes and JSON sidecar, if present, are generated from the same
  deterministic model.
- Tests prove sidecar data matches Markdown node counts/refs for fixtures.
- Markdown reverse parsing remains allowed only as compatibility fallback or
  audit, not the primary generation path.

### Phase 3 - Search Shadow Mode

Purpose: test whether Index Tree candidate narrowing can help retrieval without
changing default ranking.

Allowed file impact:

- Create private dev script such as `tools/dev/index_tree_search_shadow.py`.
- Create `tests/unit/test_index_tree_search_shadow.py`.
- Add local report under `.agent-governance/reports/index-tree-evidence-navigation/`.

Forbidden file impact:

- Do not change `life-index search` default behavior.
- Do not change `tools/search_journals/ranking.py` constants or merge policy.
- Do not change eval baselines.
- Do not expose a public flag until the recall-preservation gate passes and a
  separate public-surface decision is made.

Minimum falsifiable tests:

1. `test_shadow_mode_does_not_change_default_search_output`
   - Run the normal search path before and after enabling the private shadow
     helper in test scope; public output and order must be byte-equivalent for
     the same fixture corpus.
2. `test_shadow_candidate_set_contains_baseline_hits`
   - For every fixture query, collect baseline result paths from the existing
     search path and candidate paths from node narrowing. Assert
     `set(baseline_paths) <= set(shadow_candidate_paths)`.
3. `test_shadow_gate_fails_when_candidate_drops_relevant_hit`
   - Use a deliberately bad node filter that excludes a known relevant month.
     The gate must fail with dropped path evidence.
4. `test_shadow_mode_fails_closed_on_stale_or_missing_index`
   - If freshness is `stale` or `missing_index`, shadow mode must report
     `disabled_reason` and return no narrowing decision.
5. `test_shadow_report_is_diagnostic_only`
   - Shadow output may include candidate reduction stats, dropped-hit checks,
     and freshness status, but must not include reranked public results.

Recall-preservation gate:

```text
For the selected query fixture set:
  baseline_paths = paths returned by current default search
  shadow_candidate_paths = paths admitted by Index Tree narrowing

PASS only if:
  every baseline path is in shadow_candidate_paths;
  every stale/missing index disables narrowing;
  default search output is unchanged;
  diagnostics include dropped_path evidence on failure;
  at least one broad-recall fixture category is included.
```

Recommended fixture categories:

- exact date / month lookup;
- topic lookup using typed frontmatter;
- entity/person lookup;
- fuzzy broad recall query from existing search-quality fixtures;
- negative query where the correct result set is empty.

Tests / checks:

```bash
python -m pytest tests/unit/test_index_tree_search_shadow.py tests/unit/test_index_tree_navigation.py -q --basetemp .pytest_tmp/index-tree-shadow
python -m pytest tests/e2e/test_search_quality.py -q --basetemp .pytest_tmp/index-tree-shadow-e2e
git diff --check
```

Exit criteria:

- Shadow diagnostics prove no baseline hit was dropped in fixtures.
- Stale indexes disable narrowing.
- Default search output and ranking remain unchanged.
- Report recommends proceed/defer/drop before any public flag is proposed.

### Phase 4 - Promotion Decision

Purpose: decide whether any private prototype deserves a public contract.

Allowed file impact:

- New RFC/PRD only if Phase 1-3 evidence justifies public surface.
- Potential future public docs only after explicit ack.

Required proof:

- real consumer named;
- recall-preservation gate green;
- freshness and path privacy gates green;
- no default ranking regression;
- compatibility plan for schema/version;
- explicit no-version-bump or version-bump decision.

Exit criteria:

- Either keep private tooling, drop it, or start a new public-surface RFC/PRD.
- No silent promotion from `tools/dev` to public CLI.

## 9. Non-Goals

This RFC does not propose:

- replacing `search`, `smart-search`, `aggregate`, `analyze`, or `recall`;
- letting LLMs compute counts;
- writing persona, emotion, relationship, or narrative conclusions into Index
  Tree nodes;
- storing LLM-generated summaries as machine-trusted evidence;
- adding a public CLI command in this RFC update;
- changing `generate-index --json`;
- changing search default ranking;
- changing journal data format;
- bumping project version;
- starting social historical import;
- starting Tranche C `ai_drafting`.

## 10. Acceptance Gates For A Future Milestone

Before this becomes a production feature, a future milestone should prove:

| Gate | Required proof |
|---|---|
| Freshness | stale indexes are detected before navigation is trusted |
| Determinism | machine-trusted node fields are generated without LLM inference |
| Recall preservation | coarse narrowing does not drop entries full retrieval would need |
| Privacy | all navigation is local-first and offline-capable; paths are relative |
| Read-only safety | navigation does not write or mutate source journals |
| Evidence auditability | every derived claim can cite exact journal paths and node refs |
| Public compatibility | any public surface has schema/version and migration rules |

## 11. Risks

| Risk | Mitigation |
|---|---|
| Stale index | reuse and harden the existing `index_stale` / freshness signal before relying on navigation |
| Summary lossiness | treat indexes as coarse metadata, never as substitutes for full-text evidence |
| False compression | expose entry counts and drill-down requirements in node summaries |
| Over-trusting narrative summaries | separate deterministic metadata from any human-readable narrative text |
| LLM metadata becomes hidden truth | allow typed frontmatter only as a labeled navigation signal; exclude narrative/persona/relationship judgment |
| Markdown parser lock-in | move toward a shared deterministic builder that emits Markdown and sidecar from one model |
| Scope creep | keep Phase 1 in `tools/dev`; require new RFC/PRD for public CLI or ranking changes |

## 12. Open Questions

1. Should stable node ids remain logical (`month:2026-03`) while refs also
   preserve physical paths (`Journals/2026/03`)?
2. What minimum fixture corpus is enough for Search Shadow Mode to represent
   broad recall risk?
3. Should a future sidecar live next to each Markdown index or under a private
   `.life-index/` evidence directory?
4. Which real L3 consumer should own promotion: reflection lens, memoir
   scaffold, persona evidence bundle, or aggregate/search diagnostics?

## 13. Recommendation

Proceed to **Phase 1 Private Manifest Prototype** only.

Rationale:

- The direction has a real consumer shape: L3 reflection/memoir/persona
  evidence assembly can use node navigation without becoming core truth.
- Existing internal `tools/generate_index/navigation.py` gives enough ground to
  prototype without public API changes.
- EverOS/EverCore's useful contribution is rebuildable evidence organization,
  not its service stack or profile memory model.
- A private manifest can prove path privacy, freshness, determinism, and node
  traversal before any public contract exists.

Keep **no-version-bump** for Phase 1:

- no public CLI;
- no public API;
- no default search behavior change;
- no migration;
- no durable user-data schema change.

Do not proceed to Search Shadow Mode or public CLI until Phase 1 has a report
and an explicit next-scope decision.
